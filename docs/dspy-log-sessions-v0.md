# DSPy Experiment Log Sessions v0

The normalized DSPy experiment logs live outside the repo at:

```text
/Users/daniellerothermel/drotherm/data/code-comp/dspy-exps/v0
```

They were generated by copying local ignored artifacts from:

```text
/Users/daniellerothermel/drotherm/repos/nl-code/logs
/Users/daniellerothermel/drotherm/repos/nl-code/los
```

The source logs are copied, not moved. The original `logs/` and `los/` trees are
left in place.

## Forensic Workflow

DSPy tooling is split into three layers:

```text
logs/  ──parse_humaneval_dspy_logs.py──►  one aggregate snapshot JSON
logs/  ──sessionize_dspy_logs_v0.py────►  sessionized corpus (metadata.json + raw/)
sessionized corpus  ──inspect_dspy_* --walk──►  parsed_*_reports/
parsed_gepa_reports/  ──build_dspy_gepa_agent_bundle.py──►  agent bundle JSON
```

- `parse_humaneval_dspy_logs.py` reads a flat `logs/` tree and writes one
  combined snapshot for notebook exploration.
- `sessionize_dspy_logs_v0.py` is required before forensic inspection. It
  groups related run JSON, generation JSONL, optimization summaries, and
  optimizer internals into numbered `session_*/` directories.
- `inspect_dspy_eval_session.py` and `inspect_dspy_gepa_session.py` operate on
  session roots or a sessionized corpus root with `--walk`. Each session root
  must contain `metadata.json`.
- Raw experiment subdirectories such as `logs/eval_full_5x/baseline_direct` are
  not valid inspect inputs by themselves. They are copied under
  `session_*/raw/logs/...` during sessionization and parsed from there.

## How to Regenerate

The sessionization script is:

```text
scripts/sessionize_dspy_logs_v0.py
```

Run it from the `nl-code` repo root:

```bash
SESSIONIZE_SOURCE_ROOT=$PWD \
SESSIONIZE_OUTPUT_ROOT=~/drotherm/data/code-comp/dspy-exps/v0 \
uv run python scripts/sessionize_dspy_logs_v0.py
```

By default, the script refuses to write into a non-empty output directory. To
replace the generated `v0` directory, run:

```bash
SESSIONIZE_SOURCE_ROOT=$PWD \
SESSIONIZE_OUTPUT_ROOT=~/drotherm/data/code-comp/dspy-exps/v0 \
uv run python scripts/sessionize_dspy_logs_v0.py --overwrite
```

Then parse eval and GEPA sessions:

```bash
uv run python scripts/inspect_dspy_eval_session.py \
  ~/drotherm/data/code-comp/dspy-exps/v0 --walk

uv run python scripts/inspect_dspy_gepa_session.py \
  ~/drotherm/data/code-comp/dspy-exps/v0 --walk
```

## Directory Layout

The output root contains a manifest, numbered session directories, and one
reserved unknown bucket:

```text
v0/
  manifest.json
  session_000001/
    metadata.json
    raw/
      logs/...
  session_000002/
    metadata.json
    raw/
      logs/...
  unknown_session/
    metadata.json
    raw/
      logs/...
```

Each `session_*/metadata.json` records:

- `session_id`, `session_kind`, and confidence level
- extracted metadata such as timestamps, generation type, model, seed, counts,
  and metrics when available
- copied files with original paths, copied paths, roles, sizes, SHA256 hashes,
  and record counts when cheap to compute
- relation evidence explaining why files were grouped together
- validation notes for non-fatal mismatches, such as generation-call counts that
  differ from attempt counts

The `raw/` directory preserves source-relative paths, such as `raw/logs/...` and
`raw/los/...`, to avoid filename collisions.

## Sessionization Rules

The script groups files using explicit evidence first:

- `human_eval_dspy_run_*.json` files are grouped with generation JSONL files
  referenced by `attempts[].generation_log_file`.
- Legacy `human_eval_dspy_{direct,encdec}_eval_*.json` files are grouped with
  archive generation histories when `human_eval_dspy_snapshot_latest.json`
  records that link.
- Optimization summaries are grouped with sibling program, summary, event-log,
  and run-log files.
- Optimizer internal directories are grouped when a run log or event log
  contains an exact `log_dir=...` reference.

For four early MIPRO runs that did not preserve an explicit `log_dir=...`
reference, the script adds high-confidence inferred links based on nearby
timestamps, optimizer kind, optimization target, and saved program shape.

The `unknown_session/` directory is for files that are preserved and parsed
where possible, but not confidently attached to a numbered session.
